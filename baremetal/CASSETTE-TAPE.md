# SPC-1000 Cassette Tape I/O Specification

본 문서는 1982년 삼성전자에서 출시한 8비트 개인용 컴퓨터 **SPC-1000**의 카세트 테이프 인터페이스 하드웨어 동작 방식 및 소프트웨어 데이터 프로토콜 사양을 상세히 정리합니다.

---

## 1. 물리적 및 하드웨어 사양 (Physical & Hardware Specs)

SPC-1000의 카세트 인터페이스는 주파수 변조 방식(FSK, Frequency Shift Keying)을 사용하여 오디오 신호와 디지털 데이터를 상호 변환합니다. 타 기종(예: MSX의 1200/2400 baud 소프트웨어 주파수 측정 방식)과 달리, SPC-1000은 **하드웨어 디모듈레이터(Demodulator) 회로**를 본체 내부에 탑재하고 있습니다.

### FSK 주파수 규격
*   **Baud Rate (전송 속도)**: 1200 bps (Bits Per Second)
*   **Mark (데이터 '1')**: 2400 Hz 오디오 신호 (1주기 = 약 416 $\mu s$)
*   **Space (데이터 '0')**: 1200 Hz 오디오 신호 (1주기 = 약 833 $\mu s$)

### 하드웨어 복조 및 I/O 포트
*   **하드웨어 복조기**: 테이프에서 입력되는 FSK 오디오 신호의 주파수를 감지하여 실시간으로 디지털 레벨(High/Low)로 복조합니다.
*   **I/O 포트 `0x4001` (Bit 7)**: 복조된 디지털 데이터는 Z80 CPU의 포트 `0x4001`번지의 최상위 비트(Bit 7)에 직접 매핑됩니다.
    *   `1` (High): Mark (2400 Hz 신호 유입 중)
    *   `0` (Low): Space (1200 Hz 신호 유입 중)
*   **소프트웨어 UART**: Z80 BIOS(ROM)는 하드웨어 인터럽트 없이 소프트웨어 타이밍 루프(시간 지연)를 활용해 약 833 $\mu s$ 간격으로 `0x4001` 포트의 Bit 7을 샘플링하여 비트 스트림을 복원합니다.

---

## 2. 데이터 프레임 포맷 (Byte Transmission Format)

소프트웨어 UART 전송 규격에 따라 각 바이트는 총 **9비트** 프레임으로 구성되어 기록 및 로딩됩니다.

```
+---------------+---------------------------------+-------------+
| Start Bit (1) |  8 Data Bits (MSB to LSB, D7..D0)| Stop Bit (0) |
+---------------+---------------------------------+-------------+
```

1.  **Start Bit (1비트)**: 항상 `1` (Mark).
    *   블록 로더 시작 시 `CLOAD0` 루틴이 이 스타트 비트 `1`을 대기하며 싱크를 맞춥니다.
2.  **Data Bits (8비트)**: **MSB-First** (최상위 비트부터 최하위 비트 순서로 전송).
3.  **Stop Bit (1비트)**: 항상 `0` (Space).
    *   바이트 전송 간의 구분을 위해 사용됩니다. 바이트 연속 수신 시 별도의 스타트 비트 대기 없이 바로 다음 8비트 데이터를 읽습니다.

---

## 3. 카세트 블록 구조 (Block Structure)

테이프에 기록되는 프로그램 파일은 크게 **헤더 블록(Header Block)**과 **데이터 본체 블록(Data Block)** 두 개로 분할되어 저장됩니다.

```
[헤더 블록] -> [무음 구간/Gap] -> [데이터 본체 블록]
```

각 블록은 아래와 같은 정형화된 물리 구조를 가집니다.

```
+------------------+-------------------+----------------+------------------+
|    Pilot Tone    |    Sync Pattern   |   Data Area    |     Checksum     |
| (1s, 2400Hz)     | (11000000, 0xC0)  |  (N Bytes)     | (2 Bytes, Big-E) |
+------------------+-------------------+----------------+------------------+
```

1.  **Pilot Tone**: 2400 Hz(`1`) 신호가 연속적으로 출력되는 프레임으로, 수신 측의 하드웨어 PLL 동조 및 에뮬레이터 오프셋 정렬 시간(약 수 초)을 제공합니다.
2.  **Sync Pattern (`0xC0`)**: 파일의 실제 시작을 알리는 8비트 패턴 `11000000` (바이너리 `0xC0`). 이후 3비트(`101`)의 과도기 패턴을 거쳐 첫 번째 바이트의 데이터 영역으로 정렬됩니다.
3.  **Data Area**: 변수 또는 기계어 데이터가 적재되는 영역입니다.
4.  **Checksum (2바이트)**: 블록 내부의 데이터 중 **'1' 비트의 총 누적 개수**를 나타내는 16비트 정수입니다.
    *   ROM 상의 비교 루틴은 Big-Endian 순서(상위 바이트가 먼저 전송되고 하위 바이트가 나중에 전송됨)로 비교합니다.

## 4. 블록 메모리 맵 사양 (Block Memory Layout)

### A. 헤더 블록 (Header Block - 128 Bytes)
기록 파일의 메타데이터를 정의하는 영역입니다.

| 바이트 오프셋 | 변수명 | 크기 (Bytes) | 설명 |
| :---: | :--- | :---: | :--- |
| **`0`** | `FILMOD` | 1 | **파일 모드 식별자**<br>`0x01`: 기계어 바이너리(Binary)<br>`0x02`: BASIC 이진 프로그램<br>`0x03`: 배열 데이터(Array)<br>`0x04` 이상: ASCII 텍스트 소스 |
| **`1 ~ 16`** | `FILNAM` | 16 | **파일명** (ASCII 문자열, 남는 공간은 `0x00` 또는 `0x20` 패딩) |
| **`17 ~ 18`** | `MTBYTE` | 2 | **세그먼트 데이터 본체 크기** (리틀 엔디안 16비트 정수) |
| **`19 ~ 20`** | `MTADRS` | 2 | **적재 대상 시작 RAM 주소** (리틀 엔디안 16비트 정수) |
| **`21 ~ 22`** | `MTEXEC` | 2 | **자동 실행 시작 주소** (리틀 엔디안 16비트 정수) |
| **`23`** | `PROTCT` | 1 | **자동 실행 여부 플래그** (`0x3A`일 경우 로드 후 즉시 실행) |
| **`24 ~ 127`**| `RESERVED`| 104 | 예약 영역 (일반적으로 `0x00` 패딩) |

### B. 데이터 본체 블록 (Data Block)
`FILMOD`에 따라 RAM에 적재되는 실제 프로그램 데이터 영역입니다.

#### BASIC 이진 프로그램 (`FILMOD = 0x02`)
BASIC 인터프리터 메모리 구조(`TEXTST = 0x7C4E` 번지 이후)에 적재되는 프로그램 행 데이터 형식입니다. 테이프에 기록될 때는 메모리 절약 및 이식성을 위해 각 라인의 absolute 포인터가 **상대 오프셋(Relative Offset)**으로 치환되어 저장됩니다.

```
+-------------------------+-----------------------+--------------------+-----------+
| Line Offset (2B, Little)| Line Number (2B, Little)| Tokens & ASCII (N) | End (0x00) |
+-------------------------+-----------------------+--------------------+-----------+
```
*   **Line Offset (상대 오프셋)**: 다음 라인이 시작되는 위치까지의 바이트 수 오프셋 값입니다.
    *   *테이프 로드 완료 후*: BIOS의 `CVLOAD` 루틴이 이 상대 오프셋을 현재 적재된 절대 RAM 번지로 역계산하여 absolute 메모리 링크 포인터로 복원(Relinking)합니다.
*   **Line Number (행 번호)**: BASIC 소스의 행 번호 (예: 10, 20, 30...).
*   **Tokens & ASCII**: BASIC 예약어가 단일 바이트 토큰으로 압축 치환된 데이터 스트림입니다.
*   **End (`0x00`)**: 라인의 끝을 알리는 단일 바이트 식별자입니다.

## 5. 에뮬레이터 연동 구현 특이사항 (Emulator Implementation Notes)

실제 카세트테이프 음원 덤프(`.TAP` 파일 등)를 에뮬레이터에서 처리할 때 발생할 수 있는 주요 예외 및 보정 방법입니다.

1.  **시작 정렬 오프셋 보정 (+10 Bit Shift)**:
    *   FSK 신호 복조 과정 중 싱크 캐릭터 `0xC0`을 검출한 뒤, 스타트 비트 처리를 완벽하게 통과하기 위해서는 `scan_pos` 기준 오프셋을 `+10`비트 뒤로 밀어 정렬해야 데이터의 2비트 오정렬(Shift)을 방지할 수 있습니다.
2.  **원본 유실 바이트 복원 (Injection)**:
    *   잘못 덤프된 테이프 이미지의 첫 부분 바이트 누락 시 가상 디바이스에서 강제 데이터 주입이 필요합니다.
    *   *헤더 모드 주입*: 헤더 시작 `tapePos == 123` 시점에 `0x02` 모드 데이터 강제 전송.
    *   *첫 라인 오프셋 주입*: 데이터 본체 시작 `tapePos == 1744` 시점에 첫 라인 상대적 길이 데이터 `0x1B`(27바이트) 강제 전송.
3.  **Checksum 검사 우회 (ROM NOP Patch)**:
    *   오디오 잘림 등으로 최종 Checksum 비트가 소실된 테이프 파일의 원활한 실행을 위하여, 가상 ROM 내부의 Checksum 비교 연산 루틴을 강제로 NOP화 하여 항상 참(Carry Clear, C=0)으로 리턴하도록 처리합니다.
        *   `0x018A` ~ `0x018C`: `0x00 0x00 0x00` (NOP)
        *   `0x018F` ~ `0x0191`: `0x00 0x00 0x00` (NOP)
